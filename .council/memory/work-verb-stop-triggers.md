# Memory: Work Verb Stop Triggers

Durable semantics for how a `work` session stops. The principle: a named trigger must be a fireable trigger — no phantoms, no false guardrails.

## Decision
A `work` session has **four** stop triggers — chair-done, budget, scratch-size, and user-stop — and it ends when the first fires. Two are hard and deterministically enforceable: **budget** = `max_turns` (an exact turn count) and **scratch-size** = `scratch_max_bytes` (exact, via `wc -c`). The other two are external signals (chair-done, user-stop). **`max_tokens` was removed entirely** (not merely softened): the orchestrator has no reliable per-turn token count, so a token budget — hard *or* soft — can never fire, and worse, it was **displayed in the run banner** as `Budget: max_tokens 250k`, i.e. a spend cap the user could see but the run could never honor. By this topic's own principle (*a named trigger must be a fireable trigger*) that is a false guardrail, and a *displayed* one is worse than a merely-named one. So `max_tokens` is gone from all 8 config files, the banner, SKILL.md, and PLAN.md; token spend now rides along with `max_turns` (the run-length cap), documented as a cost expectation, not a trigger. **Wall-clock was likewise cut** as a phantom (named but with no backing config field). The `work_budget` comment lines are pinned identically across all six templates and both live `council.yaml`s; the `work` scratchpad header is pinned verbatim (`# Scratchpad — work`, chair-selected Seats line, `## Turn N — <seat>` convention, no user-input section) and anchored by a `Mode: work` worked example.
→ record: `records/20260607-220653-implement-phase-3.md` (made max_tokens soft)
→ record: `records/20260607-222944-review-phase-3.md` (review: flagged it a false guardrail)

**Resolution (done).** The review recommended demoting `max_tokens`; the user went further and **removed it outright** in favour of `scratch_max_bytes` + `max_turns` — cleaner, since there's then no phantom knob to mis-display at all. Landed on `main` alongside the Phase-3 merge. **Metric:** across real `work` runs the recorded stop reason is always one of {chair-done, max_turns, scratch_max_bytes, user-stop}, and no budget field names a number that can't halt the run.

## Why
A guardrail you can't configure or observe isn't a guardrail. Wall-clock had no config field, so naming it only misled; `max_turns` already bounds the dimension we actually care about — total iterations / work, counted exactly — so a separate wall-clock knob would add an enforcement semantic we'd have to invent for little gain. (Note the honest limit: `max_turns` is an *iteration/cost* cap, not a tight *wall-time* bound — a single turn can run arbitrarily long — so it's a loose time proxy, which is why a real `max_wall_seconds` could return in Phase 4 *if* a user shows turns are a poor time proxy.) `max_tokens` is worse than absent for the same reason squared: resting any stop on a rough, unobservable estimate fires early (wastes paid budget), late (overshoots), or never — and displaying it as "Budget" tells the user a cap exists when it can't. The honest fix in every case was to make the spec tell the truth — cut/remove the unfireable knob — not to build mechanism (e.g. mid-run token accounting) to make a word true that the user never asked for. The header/example/comment fixes exist because un-pinned formats and disagreeing fixtures drift and erode after-the-fact auditability. See [[council-record-and-memory-format]] for the format these artifacts conform to.

## Standing dissent (qa-engineer)
No live dissent — QA and PM converged on cut-wall-clock and (after the review) removing `max_tokens`. The durable tension to carry into the next change to these triggers: QA's drift / un-auditability axis (an unfireable or unpinned trigger is exactly the silent-rot failure to guard against) vs. PM's surface-area / process-overhead axis (don't ship knobs or enforcement you can't cleanly spec). Here both pointed the same way. They will diverge again when a stricter check trades off against a smaller surface — balance them then, and prefer making the spec honest over building mechanism to justify a name.
