## Recommendation

**HOLD the merge.** The branch is the right *direction* — the diff is clean, the
config unification is uniform across all 8 files, the new `Mode: work` worked
example is internally consistent and dogfoods the freshly-tightened format rules,
and cutting the phantom wall-clock trigger is the correct call. But **two of its
words currently claim an enforceability they don't have**, and this is the branch
that teaches every future `work` session what an enforceable stop trigger is — so
it must be true here. Land two cheap fixes **before** merge; two more after.

**Before merge (blocking — both are "name only what can fire"):**

1. **Demote `max_tokens` out of the stop-trigger list (QA Q2, adjudicated by PM).**
   It is named, configured in all 8 yamls, **and printed in the run banner**
   (`Budget: max_tokens 250k`, SKILL:89) — yet the orchestrator only has "a rough
   estimate" with **no defined source, cadence, or formula**, so the "lands the
   plane when exceeded" clause (SKILL:207) can never fire. A displayed spend cap
   that cannot fire is a **user-trust bug, and strictly worse than the wall-clock
   phantom the session just cut** (wall-clock had no config field — visibly
   absent; `max_tokens` looks live). By the session's own principle — *a named
   trigger must be a fireable trigger* — it is a false guardrail. Fix is the same
   medicine as wall-clock: make `max_tokens` a documented cost **expectation**,
   not a trigger; pull it off (or relabel) the "Budget:" banner so only the
   enforceable stops (`max_turns`, `scratch_max_bytes`) are shown as budget.
   **Build no token-accounting mechanism** — that is the gold-plating the council
   correctly refuses. *(Fold in the chair's F1 in the same edit: strike "bounds
   run time **precisely** / each turn is bounded wall-time" from the record and
   memory — same defect class, claiming enforceability the mechanism lacks.)*

2. **Commit the session's audit artifacts and pin the convention (QA Q1).** The
   Phase-3 record, its scratch archive, and `memory/work-verb-stop-triggers.md`
   are **untracked** — not in the branch, not committed anywhere — so the two
   conformance gates were certified against uncommitted disk files. Worse
   asymmetry: the *fictional* example's `.scratch.md` is committed while the
   *real* session's archive is not. Commit them, re-run both gates against the
   committed copies, and add to SKILL `work` step 8 (and `meeting` step 6):
   "write **and commit** the record / scratch-archive / memory." SKILL currently
   says "write the record" but never "commit it" — an unpinned convention that
   silently rots.

**After merge (non-blocking follow-ups):**

4. Add a one-line note at **PLAN §9** that Phases 1–3 landed as *spec-hardening

## Reasoning trail

- **The diff is genuinely good; the defects are in what it *claims*, not what it
  *changes*.** Verified independently: no `wall_clock`/`max_wall` references
  survive outside the audit dirs; all 8 config files carry byte-identical comment
  wording; the new example's narrative matches its scratch archive end-to-end
  (cap story, attempt counts, who-sat-out, chair-done stop reason) and both gates
  hold on it. Cutting wall-clock with no backing config is the correct,
  surface-reducing move.
- **Q2 is the load-bearing finding and it turns the session's own principle back
  on it.** The memory file states "a named trigger must be a fireable trigger —
  no phantoms" and calls a *hard* `max_tokens` "worse than absent." The session
  demoted hard→soft — but that changed the **adjective, not the observability**.
  A soft ceiling still needs an observable quantity to wrap up against; none is
  defined. So the session cut one phantom (wall-clock) and shipped another
  (`max_tokens`) in the same breath — and because `max_tokens` is still displayed
  under "Budget", it is the *more* misleading of the two. After this session
  there is **no observable cost guardrail left at all**: `max_turns × tokens-per-
  turn` does not bound total tokens (one 40-tool-call turn burns six figures
  alone), so `max_turns` is a turn-count cap, not the "runaway-cost guardrail"
  the PLAN advertised.
- **Why demote, not build (PM adjudication).** Building token accounting to
  rescue the knob is the exact gold-plating the council refuses — nobody asked
  for precise token enforcement, and `max_turns` already does ~95% of the
  runaway-cost job. The dishonesty isn't the absent mechanism; it's the
  *displayed promise*. You cut a promise you can't keep; you don't build
  machinery to keep a promise no one requested. This is the same cut/demote-over-
  build move the council has made every prior round — the twist is that a soft
  knob the user **sees** under "Budget" is not harmless the way an invisible soft
  knob would be.
- **Why Q1 blocks but differently.** The branch's deliverable (spec/code/example)
  is independently mergeable; Q1 is a defect in the *session's* auditability, not
  the branch's correctness. But an unfalsifiable gate and an uncommitted,
  unpinned audit convention are precisely the silent-rot the council guards
  against — so it's must-fix, as a parallel commit, not a reason to rebuild the
  branch.
- **The metric that says the stop-triggers are right (PM):** across real `work`
  runs, the recorded stop reason is always one of {chair-done, max_turns,
  scratch_max_bytes, user-stop}, and `max_tokens` fires **0% of the time** —
  because we stopped calling it a trigger. Every word labeled "budget" or "stop
  trigger" corresponds to something that can actually halt the run. Today the
  spec fails this by construction.
