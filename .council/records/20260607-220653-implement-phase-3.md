## Reasoning trail
QA audited all four stop triggers (chair-done / timeout-budget / scratch-size / user-stop) against the `work_budget` config across six templates and both live `council.yaml`s. Two triggers are clean (scratch-size is `wc -c`, exact; user-stop is an external signal). The defects all lived inside the **budget** trigger:

- **Gap A — wall-clock phantom (CUT).** Wall-clock was named as a budget guardrail in SKILL.md and PLAN.md but had **zero backing config field** in any template — at runtime only 3 of 4 sub-conditions could ever fire. The call was cut-vs-add. PM and chair cut it: `max_turns` already bounds total run time precisely (turns are the unit the orchestrator counts, each is bounded wall-time), so a separate wall-clock knob solves a dimension already governed while adding the one thing we keep out — an enforcement semantic we'd have to invent (sampling cadence). Adding it would have been six new config surfaces to govern an already-bounded dimension: gold-plating. Wall-clock *is* cheaply observable, so it is logged as a Phase-4 re-add candidate **if** a real user reports `max_turns` is a poor time proxy.

- **Gap B — `max_tokens` hard-vs-soft (made SOFT).** Three sources gave three readings (yaml comment "soft ceiling," SKILL.md hard stop, PLAN.md "hard stop"), and the spec itself admits only "a rough token estimate" exists. The hard-vs-soft call resolved to soft because you cannot deterministically hard-stop on an unobservable estimated quantity — a "hard" max_tokens fires early, late, or never, i.e. a *false* guardrail (QA's highest-severity finding). The fix flows toward the templates' existing truth: the only two enforceable hard stops are `max_turns` and `scratch_max_bytes`; `max_tokens` is a chair-discretion "land the plane" nudge. Building real mid-run token accounting to make "hard" honest was explicitly rejected — engineering to honor a word the user never asked for.

- **Gaps C/D/E — completeness/anti-drift.** The `work` scratchpad header was underspecified ("task header" only) where the meeting header is pinned verbatim; there was no `Mode: work` worked example (every fixture was `meeting`); and the `work_budget` comment lines disagreed between the canonical example and the shipped templates on which caps are "hard" — quietly reintroducing Gap B through the back door. All three were closed in templates/spec/examples.

QA and PM **converged** on both load-bearing calls (cut-A, soften-B) with no conflict. Both held the standing rule: named ≠ enforceable is a spec bug, and the fix is to make the spec tell the truth, not to build mechanism to make the words true — so all five fixes are template/spec/example completions and **no Gate 3 was added**. Meta-observation, consistent with Phase 2: the verb already existed and was sound; the value of the session was hardening the spec against drift and removing a guardrail that couldn't fire.

## Dissents (preserved)
This was convergence, not conflict — there is no live disagreement to record, and I won't manufacture one. QA and PM reached the same cut-A/soft-B conclusion from different starting axes and the chair locked it.

What travels with this work is the **standing QA↔PM tension** on every format/spec change: QA's axis is drift and un-auditability (a named-but-unfireable trigger, or a header that isn't pinned, is exactly the failure mode that erodes after-the-fact checkability); PM's axis is process overhead and surface area (don't ship knobs or gates whose enforcement you can't cleanly spec). Here the axes pointed the same way — cutting the phantom and softening the lie *reduce* both drift and surface — so the tension was latent, not active. It will reappear on the next change that pits a stricter check against a smaller surface.

QA's verification caveat, recorded honestly: the PASS came with one pre-existing nit out of the diff's scope — the *meeting* scratch fixture still said the scratchpad was "Deleted," contradicting the load-bearing archive-not-delete invariant. The chair folded the one-line fix in (`dabc9a6`) rather than leaving two fixtures disagreeing on an invariant Gate 1 depends on; with that, no open nits remain.
