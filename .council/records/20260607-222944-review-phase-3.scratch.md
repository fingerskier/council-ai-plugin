## Turn 1 — staff-engineer (chair)

**Verdict: endorse the merge — the work is correct and the diff is clean — but the central justification is overstated in a nameable way. Fix the prose, not the decision.**

**F1 (should-fix, durable) — the wall-clock cut is right; its stated REASON is dishonest.** The decision is sound (a guardrail with no backing config can never fire; cutting beats inventing 6 config surfaces + a sampling-cadence semantic). But the record (`records/...phase-3.md:18`) and memory (`memory/work-verb-stop-triggers.md:6,10`) justify it with "`max_turns` already bounds run time **precisely** ... each turn is bounded wall-time." That's false: a single autonomous seat turn is one subagent invocation that can fan out into arbitrary tool calls + a long generation — **no upper bound on elapsed minutes.** `max_turns` is an *iteration/cost* proxy, exact about turn count, NOT about time. The tell that the authors half-know this: their own Phase-4 re-add condition re-admits wall-clock "if a real session shows `max_turns` is a poor wall-time proxy" — you can't call it *precise* and hold a contingency for it being *poor*. Those contradict. Honest framing: cut because (a) no backing config, and (b) `max_turns` already bounds the dimension we care about — total work/cost — making a second time-knob redundant surface. Strike "precisely / each turn is bounded wall-time" from the record + memory. It's the one place this work claims more than it can defend, and it now lives in memory.

**F2 (nice-to-have) — "timeout" is now a vestigial label.** The trigger is still named "timeout / budget" (PLAN.md:172, SKILL.md:187), but "timeout" WAS the wall-clock component that got cut — it now has no referent. The trigger is real as *budget* (max_turns is hard); "timeout" is a ghost of the deleted dimension. Rename to "budget" and the four-trigger story is fully self-consistent. (The count "four" is otherwise honest — chair-done / budget / scratch-size / user-stop are categories, not config fields; budget having one hard knob, max_turns, is fine since scratch-size is its own separate trigger.)

**F3 (follow-up, separate change) — scope-relabel pattern, 3rd time, now a misleading PLAN.** Phases 1/2/3 each concluded "the verb already existed; the session hardened the spec." Not deception — it's the true artifact of an MVP that front-loaded architecture into the spec before phasing was written. But §9's headers still read as build-phases when the delivered work 3× running has been conformance-hardening; a reader planning remaining phases will mis-estimate what's left (think mechanism, get spec-truing). Fix: one-line note at PLAN §9 that Phases 1–3 landed as hardening passes over a front-loaded spec, so remaining phases read the same. Out of scope for this branch.

**Verified clean:** branch tree has no `wall.clock`/`wall_clock`/`max_wall` outside the audit dirs (where the cut is documented, not referenced as live).
