## Round 1 — staff-engineer

I read SKILL.md, commands/council.md, the templates/personalities, the Codex manifests, the tests, README, the examples, and all three memory topics plus the publish-readiness record. Verdict up front: **the architecture is sound.** Personalities-as-data, one orchestrator, chair-routes-and-synthesizes, two-tier memory manifest, five real stop triggers, two mechanical gates — the core deliberation design holds together and the dogfood records prove it runs. What's left is not structural; it's operational edges and doc hygiene. Findings, ranked:

**Deliberation logic — spec gaps with real failure modes**

1. **Chair routing in `work` is under-specified.** SKILL.md says the chair "is spawned the same way as any seat" and its task is to route, and step 6a says the chair decides who acts next — but it never pins whether each routing decision is a *chair spawn* (doubling spawns per turn, invisible to `max_turns`) or the orchestrator judging inline as the chair. Worse for auditability: the per-turn sub-goal is injected into the seat's prompt but is **not required to be written to the scratchpad**, so the chair's routing intent is absent from the very audit trail the gates protect. Fix: pin the mechanism, and require a one-line chair entry per turn (who's next, sub-goal, why) in the scratchpad.

2. **No interrupted-session story.** A crash or user abort mid-session leaves an orphaned `scratch/<id>.md` (gitignored, never surfaced again) and, for `work`, a dangling worktree and branch. Nothing detects or offers to resume/archive them; `info` is blind to them. The scratchpad *is* the recovery artifact — we just never use it. Cheap spec fix: on preflight, if a scratchpad exists with no matching record, say so and offer resume or archive; have `info` list dangling scratchpads/worktrees.

3. **Audit-trail commit scoping is unpinned.** Both verbs end by committing record + archived scratch + memory "on the current branch." If the user's tree is dirty — which no step forbids — an orchestrator that reaches for `git add -A` sweeps the user's uncommitted work into a council commit. Pin the exact paths (`git add .council/records/<id>* .council/memory/`). Same defect in work step 6b: "main working tree stays clean" false-alarms on a legitimately dirty tree — the check should be "no *new* changes vs. session-start snapshot."

4. **Meetings are unbounded and the user steers blind.** `work_budget` applies only to `work`; a meeting has no round cap and no scratch bound, and each round re-injects the full scratchpad into every seat — cost grows quadratically in rounds. That's fine *given* the human is the stop trigger, but the human gets no signal. Per the stop-trigger principle in memory (a named trigger must be fireable), don't add a fake budget — add a **signal**: put round number and scratchpad size in the round-summary header ("Round 3 · scratchpad 41 KB") so Conclude-vs-Continue is an informed choice.

5. **Seat-spawn failure is unhandled.** If one seat errors mid-round, the spec doesn't say whether the round continues, retries, or dies. One line fixes it: note the failure under that seat's heading, continue the round, skip a twice-failed seat for the session.

**UX**

6. `work` reports only at the end. Pin a one-line per-turn progress report ("Turn 3/12 — qa-engineer: regression tests for the retry helper") — the orchestrator already tracks all three values. Biggest trust win available for the autonomous verb.

7. Round order anchors every round on the same first seat (roster order). Rotating the starting seat per round is a zero-mechanism de-biasing tweak — PM/QA should weigh whether the anchoring is real enough to bother.

**Workflow / hygiene**

8. **Follow-ups are write-only.** Records accumulate `- [ ]` items with owners (publish-readiness has five open) and no verb ever re-reads them. Either surface open follow-ups in `info` or the preflight manifest, or stop pretending they're tracked.

9. **The gates are specified as scriptable but no script ships.** Gate 2's enum was designed so closure is "a script, not a skim" — yet `test/` only checks manifests and template→seat references. A conformance test running both gate checks over `examples/sample-council/` and the dogfood `.council/` would make the fixtures self-verifying; fixture/spec drift has already happened once. QA's lane; I endorse it as the highest-value single addition.

10. **README duplication.** The "Why a council?" section ends with a verbatim repeat of the intro paragraph, and "Getting started §2" vs "## Install" are two overlapping install narratives. Given the v0.1 bar was "a stranger passes a cold read of the README," a doubled README undercuts our own gate. Also minor: SKILL.md states the stop-trigger semantics twice in full (step 4 and step 6c, ~40 lines each) — state once, reference once.

None of this reopens settled decisions — no new budget knobs (PM's surface-area line holds), no third gate, and security's standing injection dissent stays parked per publish-readiness. Suggested pickups: **security-engineer** — items 2/3 (orphaned branches and commit-sweep are integrity surfaces); **qa-engineer** — item 9 plus testability of items 1/3; **product-manager** — items 4/6/8 and whether README cleanup blocks the v0.1 cold-read.

## Round 1 — security-engineer

I took the routed items (2 and 3), threat-modeled the full session lifecycle around them, and re-read my standing dissent in `memory/publish-readiness.md` before leaning on it. It stays parked: the injection surface (seat/memory files verbatim into bash-capable workers) is documented, accepted for v0.1, and owed the `<untrusted>` delimiter work before councils become shareable. Nothing below reopens it — but three of my findings shrink the blast radius of that same channel cheaply, which is the right posture while the dissent is parked. Ranked by exploitability x blast radius:

**1. Must-fix — staff's items 3 and 5/6b are one attack chain, and the spec's own text shows the habit.** The only pinned `git add` in SKILL.md is the worktree commit (`git -C .council/worktrees/<id> add -A` — safe, `-C`-scoped to the isolated tree). The audit-trail commits (meeting step 6, work step 9) pin *no command at all* — "then commit the record, the archived scratch, and the memory files." And this isn't hypothetical orchestrator behavior: our own publish-readiness scratch archive records the near-miss in so many words — untracked `skidl.log`/`skidl.erc` carrying PII where "next `git add -A` sweeps them in." Now chain it: a repo-carried hostile memory topic or persona (the parked dissent's exact premise) steers a work seat to write outside the worktree — the worktree boundary is prompt-only; step 6b's "main working tree stays clean" check names no failure action, so an orchestrator that notices has no instruction, and one that doesn't notice proceeds; the unpinned audit commit then sweeps the escaped payload — or simply the user's dirty tree, including any temporarily-pasted token in a tracked config — into a legitimate-looking council commit on the user's branch. No merge decision ever happens; the payload is already landed. Three one-line breaks, each independently worth having: (a) pin the audit commit as an explicit allowlist — `git add .council/records/<id>.md .council/records/<id>.scratch.md .council/memory/` — never `-A`, never `-am`; (b) give the step-6b check a failure action: halt the loop, report the escaped paths, never commit them; (c) staff's snapshot-diff so the check means "no *new* main-tree changes." The settled auto-commit-on-current-branch decision is only safe *because of* (a) — pinned paths are what make an unattended commit on the user's branch acceptable at all.

**2. Must-fix before heavy `work` use — the archive pipeline auto-commits whatever seats said, with no leakage guard.** Live `scratch/` is gitignored, but conclude renames it to `records/<id>.scratch.md` and commits it — mandatory, and in `work` fully autonomous, no human between a seat's words and the commit. Concrete: a work seat reads a config or `.env` inside the worktree, quotes a connection string in its turn report; conclude → archive → commit → the secret is in history, and on the next push, on the remote. SKILL.md contains zero mention of secrets anywhere in the pipeline. Two cheap layers: one line in the seat prompt template ("reference file paths; never quote credential or secret values into the scratchpad"), and a commit-step check — grep record + scratch archive for obvious secret shapes before the audit commit. To be explicit for PM: this is **not a third synthesis gate** — the two-gate line holds. It's commit hygiene, living in the same commit step as the pinned pathset from finding 1; if QA's conformance script (staff item 9) lands, that script is its natural home.

**3. Hardening, not v0.1-blocking — make meeting read-only a capability, not a request.** The spec is honest that the meeting's read-only contract is *injected into the prompt* — which means it's enforced on the same channel an injected persona controls. On Codex we already differentiate: `worker` agent type for work turns, default type for read-only meeting turns. On Claude, every seat in both verbs is `subagent_type: general-purpose` — full tools. Mirror the Codex distinction: spawn Claude meeting seats with a read-only agent type where the host offers one. That converts the meeting verb's guardrail from persuasion to capability at zero UX cost, and it partially defuses the parked dissent for meetings specifically — a hostile persona in a meeting seat could no longer mutate anything. Honest limit: it stops mutation, not disclosure; a hostile persona can still read files and exfiltrate content into the scratchpad/record. Still worth doing.

**4. Endorse staff item 2 (orphan story) with two security riders.** (i) A crashed `work` session can leave a *committed branch with no record* — which quietly breaks the invariant everything else here defends: every council-authored change traces to an audit record. Recovery must label such branches **unaudited** and offer archive-or-delete, not leave them looking mergeable; a user who finds a stale `council/work-*` branch six weeks later should be told there is no record behind it. (ii) On resume, the recovered scratchpad is untrusted seat output like any live one (it's already named in the deferred `<untrusted>` wrap list — no new boundary), and header fields the orchestrator *acts on* — `Started (epoch)` arms the wall-clock trigger — must be parsed from the header block only, never last-match, since seats append to this file and an appended forged header line must not win.

Brief endorsements in others' lanes: staff item 1 (chair routing line in the scratchpad) is audit-trail completeness — the gates only protect what gets written, so routing intent belongs in the artifact; and item 9's gate script makes the audit trail mechanically verifiable, which is the property findings 1 and 2 depend on.

No dissent from the council's direction this round. One triage line for the chair: findings 1(a) and 1(b) are the only items I'd call blocking for continued *dogfood* use, not just publish — they guard the user's own branch, today, on this repo.
