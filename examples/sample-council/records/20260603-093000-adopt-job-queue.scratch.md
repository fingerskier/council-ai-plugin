## Round 1 — staff-engineer
Request handlers are timing out on bulk exports — the trigger for this whole
question. We already operate Redis for sessions, so BullMQ is the low-friction
option: retries, delayed jobs, and a dead-letter queue with no new infra to
learn. The real alternatives are a managed queue (SQS) or a DB-table-as-queue.
Framing the round: is async work justified, and if so does reusing Redis beat
adopting new infra?

## Round 1 — security-engineer
Agreed async is needed, but note what changes: the moment Redis backs a queue it
stops being a cache and becomes a *correctness* dependency. The sessions Redis is
provisioned as an ephemeral cache — no persistence guarantee, and I'd want to
re-check auth. A `FLUSHALL` or an evicted key silently drops in-flight jobs.
Dissent (not blocking the decision, blocking the *launch*): gate go-live on a
persistent, auth'd Redis, separate from the cache instance.

## Round 1 — qa-engineer
Retries are only a feature if the jobs are idempotent — otherwise a redelivered
webhook double-charges or double-emails. Dissent: no merge of any job type until
it has a test proving idempotency under double-delivery, plus dead-letter
handling. I don't object to BullMQ; I object to shipping it without that harness.

## Round 1 — product-manager
Exports are the loudest user pain and the lowest blast radius if a job is dropped
mid-rollout — start there. SQS is real infra and egress cost for a problem the
Redis we already run can solve; defer it unless volume forces the issue. Ship
behind a flag, cut over one job type at a time.

## User input after Round 1
**Conclude** — no new steer; direction (BullMQ on existing Redis, exports first,
security + QA gates preserved) is clear. Chair to synthesize.
